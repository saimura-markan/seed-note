# Seed Note 思想書

> このドキュメントは設計書ではない。  
> Seed Note がなぜ存在するのか、何を目指しているのかを記した「思想書」である。  
> 将来このシステムを開発するすべてのエンジニアと AI は、まずこのファイルを読むこと。

---

## 1. Seed Note とは

Seed Note は、清掃・解体・産廃業界の現場で発生するクレームを記録・分析・共有するためのシステムである。

しかし単なる「クレーム管理ツール」ではない。

クレームという出来事を、組織が成長するための情報として捉え直し、個々の現場で起きた出来事が組織全体の知恵になるまでの一連のプロセスを支援することを目的として作られた。

**なぜ作られたのか。**

現場の最前線では、毎日何らかのミスやクレームが発生する。その多くは「担当者が気をつけなかった」「その人の問題だった」として処理され、記録も残らず、再び同じことが起きる。改善されない。共有されない。繰り返される。

このループを断ち切るために Seed Note は存在する。

---

## 2. 基本理念

## 「クレームは、成長の種。」

この言葉は、クレームに対する根本的な態度の転換を表している。

クレームは通常、「起きてしまった悪いこと」として扱われる。担当者は責められ、謝罪し、その場を収めることに集中する。そして終わる。

Seed Note はこれを拒絶する。

クレームは、組織の中に潜む問題が表面に現れた瞬間である。それは問題の「発生」ではなく、問題の「発見」である。発見できたということは、対処できるということだ。対処できるということは、仕組みを変えられるということだ。

つまりクレームは、組織が自分自身の課題を知るための最も直接的な情報源である。

「種」という言葉には、可能性が宿っている。種はそのままでは何も生まない。しかし土に埋め、水をやり、育てれば、やがて実を結ぶ。クレームもまた、正しく記録し、分析し、共有することで、組織の成長に変わる。

## 3. Seed Note が解決したい問題

### 3-1. クレームが個人の責任追及で終わる

「誰がやったのか」「なぜあの人はできなかったのか」という問いは、犯人を特定することはできても、再発を防ぐことはできない。同じ状況に別の人が置かれれば、同じことが起きる。

個人を責めることは、問題の本質から目を逸らすことである。

### 3-2. 同じミスが繰り返される

個人責任で終わると、仕組みは変わらない。環境は変わらない。だから同じミスが、同じ部署で、あるいは別の部署で、また起きる。記録がなければ「また起きた」とさえ気づかない。

繰り返しは、記録のなさと仕組みの欠如の産物である。

### 3-3. 部署間で学びが共有されない

ある現場で起きたクレームから得た教訓が、別の現場に届かない。同じ問題が複数の現場で独立して発生し、それぞれが独自に対応し、誰も全体像を知らない。

組織として学ぶためには、個人の経験を組織の知識に変換する仕組みが必要である。

### 3-4. 原因ではなく現象だけで終わる

「お客様が怒っていた」「作業が雑だった」という現象の記録だけでは、何も変わらない。なぜ怒らせたのか、なぜ作業が雑になったのか、その背景にある構造的な原因を掘り下げなければ、真の改善は起きない。

現象の記録は出発点であり、目的地ではない。

## 4. クレームに対する考え方

Seed Note は「犯人探しのシステム」ではない。

これは単なるスローガンではなく、システム設計の根幹に関わる原則である。

クレームが発生したとき、最初に問うべきは「誰が悪かったのか」ではなく「なぜこのシステムはこれを防げなかったのか」である。人が失敗するのは、その人が無能だからではない。多くの場合、失敗しやすい環境や仕組みの中に置かれているからである。

Seed Note は「**組織学習のシステム**」である。

担当者が記録し、管理者が分析し、事業責任者が真因を掘り下げ、役員が承認する。この一連のプロセスは、一人の人間を裁くためではなく、組織全体で「なぜこれが起きたのか」「どうすれば防げるのか」を考え抜くための構造である。

承認完了した案件が掲示板に公開されるのも、「このような失敗があった」と晒すためではない。「このような学びを組織全体で共有する」ためである。

---

## 5. 真因分析の考え方

### 5-1. 真因分析の目的

Seed Note における真因分析の目的は、以下の3つである。

**① 再発防止**  
同じクレームが再び起きないよう、直接原因の根本にある構造的な問題を特定し、対策を打つ。

**② 未然防止**  
蓄積されたクレームのパターンから、まだ表面化していない潜在的なリスクを検知し、問題が起きる前に手を打つ。これは Seed Note が将来的に目指す姿であり、現時点では掲示板とデータ蓄積がその基盤となっている。

**③ 仕組み改善**  
個別の対策で終わらず、再び同じ環境が生まれないよう、体制・ルール・教育・管理の仕組みそのものを変える。

### 5-2. なぜソクラテス式対話を採用するのか

Seed Note は、改善報告書と深掘り分析の2箇所でソクラテス式対話を実装している。

これは「入力させるため」ではない。「考えさせるため」である。

人は問いを与えられることで、自分では気づかなかった深い理由に到達することがある。「なぜ起きたのか」「なぜ防げなかったのか」という問いを繰り返すことで、現象から原因へ、原因から構造へと思考が深まる。

ソクラテス式対話は、分析の質を高めるための設計的な選択である。

### 5-3. 真因カテゴリー 8分類の考え方

真因を特定した後、それがどの種類の問題に属するかを分類する。現在の8分類は、組織に潜む構造的な問題のパターンを表している。

| カテゴリー | 意味するもの |
|-----------|------------|
| 教育不足 | 知識・技術・姿勢の伝達が不十分だった |
| 標準化不足 | やり方が人によって異なり、品質が安定しない |
| ルール未整備 | 何をすべきかが明文化されておらず、判断が属人化している |
| システム不備 | 管理ツール・確認の仕組み自体に欠陥がある |
| 顧客確認不足 | 顧客との認識合わせが事前に行われなかった |
| 引継ぎ不足 | 情報が次の担当者・部署に伝わらなかった |
| マネジメント不足 | 管理者による適切な指導・確認がなかった |
| 人員配置問題 | 適切なスキルや経験を持つ人が配置されていなかった |

この分類は「誰の責任か」を示すものではない。「どの仕組みを変える必要があるか」を示すものである。

「教育不足」と分類されたとき、責任を問うべきは「教育しなかった個人」ではなく「教育の仕組みが機能していなかった組織」である。

## 6. 役割

### 担当者（admin / judgment / user）

現場の最前線に立ち、クレームと向き合う人。お客様への連絡、作業者からの聞き取り、是正案の提出、改善報告書の作成を担う。

担当者に求められるのは「正直な記録」である。何が起きたか、何を考えたか、何を試みたかを飾らず記録することが、後の分析の精度を決める。

担当者は裁かれる対象ではない。組織学習の起点となる情報提供者である。

### 管理者・主任（manager）

現場と経営の橋渡し役。担当者から上がってきた是正案を確認し、承認または差し戻しを判断する。

管理者の役割は「正しいかどうかを判定する」ことではなく、「現場の対応が本質的な改善につながっているかを見極める」ことである。担当者の思考が現象止まりになっていると感じたら、深掘りを求める権限を持つ。

### 事業責任者（director）

深掘り分析を担う役割。改善報告書の内容を踏まえ、ソクラテス式対話を通じて真因を掘り下げ、真因カテゴリーを分類し、横展開と真因対策を策定する。

事業責任者は「このクレームがなぜ起きたか」を個別に考えるのと同時に、「この組織にはどのような構造的問題があるか」という視点で分析することが求められる。

深掘り分析の質が、Seed Note の価値の中心を担う。

### 役員（executive）

役員の役割は「個別対応の最終判断者」ではない。

役員が承認するのは、担当者・管理者・事業責任者が作り上げた「組織としての学び」の総体である。役員は「この真因分析と対策は、全社的な再発防止につながるか」という視点で承認判断を行う。

つまり役員は「**個別のクレームを裁く役割**」ではなく、「**全社で再発防止できる仕組みを作る役割**」である。

役員承認によって掲示板に公開された案件は、全社員が学べる組織の財産となる。役員はその財産の質を担保する最後の審査者である。

## 7. Seed Note が目指す未来

### 現在（フェーズ1：記録と学習）

まず「記録する」ことが基盤である。クレームが起きたとき、何が起きたかを正確に記録する。誰が対応したか、何を考えたか、どう改善したかを残す。この記録なしに、学習は始まらない。

掲示板は、この記録を組織全体で共有するための仕組みである。承認完了した案件が蓄積されることで、「うちの会社にはどんなクレームが多いか」「どの真因カテゴリーが頻発しているか」が可視化される。

### 次のフェーズ（フェーズ2：傾向分析）

蓄積されたデータから、傾向を読む。

- どの部署でどのカテゴリーのクレームが多いか
- 同じ真因が繰り返されていないか
- 対策の進捗はどうか

これにより、個別のクレーム対応から「組織全体の弱点」の把握へと視野が広がる。

### その先（フェーズ3：未然防止と AI 分析）

最終的に Seed Note が目指すのは、クレームが起きてから対応するシステムではなく、クレームが起きる前に予兆を検知し、事前に手を打てるシステムである。

`@anthropic-ai/sdk` はすでに導入済みである。これは将来の AI 分析機能を見越した準備である。蓄積されたクレームデータと真因カテゴリーをもとに、AI が傾向を分析し、潜在的なリスクを指摘する仕組みが構想されている。

未然防止とは、クレームが「起きてしまうもの」という前提を疑うことである。正しい仕組みと継続的な学習があれば、クレームの発生自体を減らすことができる。それが Seed Note の究極の目標である。

## 8. 開発方針

Seed Note を将来開発するすべての AI とエンジニアへ。

### 守るべき原則

**① 個人責任追及の方向へ進めない**  
「誰がやったか」を明確にする機能、特定の個人を責める UI、担当者を監視する仕組みは、Seed Note の思想と相容れない。仕組みの改善に資さない機能は追加しない。

**② 組織学習を最優先する**  
新しい機能を追加するとき、「これは組織全体の学習につながるか」を問うこと。個人の利便性より、組織知識の蓄積と共有を優先する。

**③ 真因分析を重視する**  
現象の記録だけで終わる機能拡張は本質的でない。「なぜ起きたか」「どうすれば防げるか」を掘り下げる機能こそが Seed Note の核心である。ソクラテス式対話の質を下げる変更は慎重に行う。

**④ 再発防止より未然防止を目指す**  
再発防止は「起きた後の対応」である。未然防止は「起きる前の対応」である。システムの進化は常に「次の問題が起きる前に気づける仕組み」の方向に向かうべきである。

### やってはいけないこと

- クレームを「件数」「達成率」「スコア」として評価する機能（個人を数値で裁くことにつながる）
- 担当者の対応速度だけを強調する UI（プレッシャーではなく、品質を重視する）
- 承認者の承認パターンを分析して個人の判断傾向を可視化する機能
- 真因分析を省略・簡略化するショートカット

### 変えてよいこと

- UI のデザインと使いやすさの改善
- 掲示板の検索・分析機能の強化
- AI 分析機能の実装（真因傾向の可視化・予兆検知）
- 通知・リマインダー機能
- モバイル対応

---

## おわりに

Seed Note は、技術のためのシステムではない。現場で働く人のためのシステムである。

クレームという、誰もが避けたいと思う出来事を、誰もが向き合える仕組みに変えること。そしてその向き合いの蓄積が、組織を少しずつ、確実に強くしていくこと。

それが、「クレームは、成長の種。」という言葉に込められた意味であり、Seed Note が存在する理由である。
