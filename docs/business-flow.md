## 2. ソクラテス式対話フロー（改善報告書 / 現象原因）

`CorrectionSubmit.jsx` に実装。

```
[直接原因テキストを入力]
        │ useEffect: directCause 変更を検知
        │ ※ DB ロード時は自動起動しない（loadingFromDb.ref で制御）
        ▼
  phase: 'q1'
  「その原因はなぜ起きたのでしょうか？」
        │ 回答入力 → 入力ボタン
        ▼
  phase: 'q2'
  「では2度と起きないようにするには何が必要だと思いますか？」
        │ 回答入力 → 入力ボタン
        ▼
  phase: 'final_check'
  「本当にそれで2度と起きませんか？」
        │
   ┌────┴────┐
  はい      いいえ
   │         │
   │     phase: 'retry'
   │     「では何が足りないと思いますか？」
   │         │
   └────┬────┘
        ▼
  phase: 'complete'
  回答内容を「④ 運用改善案」テキストエリアに自動反映

[DB 保存]
  complaint_corrections の socratic_answers（jsonb）に保存:
  { q1: "...", q2: "...", retry: "..." }

[再提出時の復元]
  fetchData 時に socratic_answers を読み込み、
  socrPhase='complete' にセットして過去回答を表示
```

## 3. 深掘り分析フロー（真因 / DeepAnalysisForm）

`DeepAnalysisForm.jsx` に実装。

```
[ソクラテス式対話：真因]
  phase: 'q1'
  「なぜ今回のようなことが起きたと思いますか？（仕組みや体制の観点で）」
        │
        ▼
  phase: 'q2'
  「それはなぜ会社の仕組みとして防げなかったのでしょうか？」
        │
        ▼
  phase: 'final_check'
  「本当にそれが根本的な原因だと思いますか？」
        │
   ┌────┴────┐
  はい      いいえ
   │         │
   │     phase: 'retry'
   │     「では何が足りないと思いますか？」
   │         │
   └────┬────┘
        ▼
  phase: 'complete'
  対話内容を「真因」テキストエリアに自動反映

  ※ rootAnswers（対話回答）は現在 DB に保存されていない（未実装）
  ※ ページリロードで回答は消える

[フォーム入力]
  ・真因（ソクラテス対話から自動入力、手動修正可）
  ・組織改善案（必須）
  ・真因カテゴリー（8分類から1つ選択、必須）
  ・真因詳細（必須）

[横展開]
  ・対象部署（チェックボックス、複数選択可、1つ以上必須）
  ・周知内容（必須）

[真因対策]
  ・担当者（必須）
  ・期限（日付、必須）→ 残日数カウントダウン表示
  ・進捗（未着手 / 進行中 / 完了）

[送信条件（canSubmit）]
  rootCause, orgImprove, rootTheme, rootDetail,
  horizDepts（1件以上）, horizContent,
  actionAssignee, actionDeadline
  → 全て入力済みで送信可能
```

## 4. 真因カテゴリー 8分類

`DeepAnalysisForm.jsx` の `ROOT_THEMES` 定数より確認。

| # | カテゴリー | 意味（コメントなし・推測含む）|
|---|-----------|--------------------------|
| 1 | 教育不足 | 作業者への教育・訓練が不足 |
| 2 | 標準化不足 | 作業手順・チェックリストが整備されていない |
| 3 | ルール未整備 | 社内ルール・規程がない |
| 4 | システム不備 | 管理システム・ツールが機能していない |
| 5 | 顧客確認不足 | 顧客との事前確認が不十分 |
| 6 | 引継ぎ不足 | 情報伝達・引継ぎが行われていない |
| 7 | マネジメント不足 | 管理者による監督・指導が不足 |
| 8 | 人員配置問題 | 適切な人員配置がされていない |

**注意：** `Approval.jsx` の `ROOT_THEME_COLORS` は旧4分類（`標準化不足・教育不足・報告不足・顧客視点不足`）のまま残存しており、現行の8分類と不整合。集計グラフが正しく機能しない可能性がある。

## 5. ステータス一覧と遷移

| ステータス値 | 発生タイミング | 遷移元 | 遷移先 |
|-------------|-------------|-------|-------|
| `受付済` | ComplaintNew 登録時 | — | 対応中 |
| `対応中` | 初回 contact ログ記録時 | 受付済 | 是正案提出 |
| `是正案提出` | 対応判断→上司報告時 | 対応中 / 是正案差し戻し | 是正案承認 / 是正案差し戻し |
| `是正案差し戻し` | DeepAnalysisForm 否認時 | 是正案提出 | 是正案提出 |
| `是正案承認` | DeepAnalysisForm 承認時 | 是正案提出 | 改善報告書提出 |
| `改善報告書提出` | CorrectionSubmit 提出時 / judgment='手直し'承認時 | 是正案承認 / correction_rejected | 深掘り提出 |
| `correction_rejected` | DeepAnalysisForm で改善報告書否認時 | 改善報告書提出 | 改善報告書提出 |
| `深掘り提出` | DeepAnalysisForm 深掘り提出時 | 改善報告書提出 | 承認完了 |
| `承認完了` | Approval で全員承認時 | 深掘り提出 | （終了） |
| `周知完了` | **未実装**（ProgressBar 上のみ定義） | — | — |

**注意：** `correction_rejected` は `STATUS_ORDER` 配列に含まれていない。ダッシュボードのステータスフィルターに表示されない可能性がある。

## 6. 対応期限タイマー

クレーム受付時の感情レベルによって対応期限が自動設定される。

| 感情レベル | ラベル | 対応期限 |
|-----------|-------|---------|
| Lv.1 | 穏やか | 60分 |
| Lv.2 | 気になる | 60分 |
| Lv.3 | 注意 | 60分 |
| Lv.4 | 緊張 | 30分 |
| Lv.5 | 最高緊張 | 15分 |

- タイマーはダッシュボードの `ComplaintCard` 内に表示（1秒更新）
- 期限超過時は赤表示・「超過」表示
- 初回連絡ログがある場合は「初回連絡まで○○:○○」に切り替わる（タイマー停止）

## 7. 掲示板（bulletin_board）自動投稿フロー

```
Approval.jsx の handleApproval()
        │
        │ 全員の status === 'approved' を確認
        │
        ├── complaints.update(status='承認完了')
        │
        └── bulletin_board の重複確認
            （complaint_id で maybeSingle()）
                │
           ┌────┴────┐
         存在する    存在しない
           │              │
         スキップ      bulletin_board.insert({
                          complaint_id,
                          content: {
                            site_name, description, received_at, assignee,
                            contact_logs[], hearing,
                            correction_action, direct_cause, improvement,
                            root_cause, root_theme, root_detail, org_improvement,
                            action_assignee, action_deadline, action_progress,
                            horizontal_departments[], horizontal_content
                          }
                        })
```

**content フィールドは jsonb 型。全情報を1オブジェクトにパックして保存。**
